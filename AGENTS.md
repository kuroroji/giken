# AGENTS.md — くろろじ技研 エージェント常時ルール

このリポジトリは、ゲーム発掘YouTuber「くろろじ」の公式サイト(https://kuroroji.site)。
Astro 5 + TypeScript + Tailwind CSS v4。静的生成・React不使用。
プロジェクト構成・デザイン決定事項・仮置き事項の詳細は `CLAUDE.md` を参照(併存させること。削除・統合しない)。

記事作成の詳しい反復手順はスキルに分離してある。

- 記事の新規執筆・リライト(相談フェーズ→執筆→検証→公開) → `.codex/skills/article-writing/SKILL.md`
- ゲーム紹介記事の標準処理(ストアボタン・画像・表) → `.codex/skills/game-article/SKILL.md`

以下は、作業内容にかかわらず常に守るルール。

## 並行作業ルール(Claude CodeとCodexの共同作業)

このリポジトリは Claude Code と Codex の両方が作業する。

- **作業開始時とプッシュ前に必ず `git pull origin main` する**。見覚えのないコミットがあっても異常ではない(もう一方のエージェントの作業)
- ルール・手順に変更があったら、`AGENTS.md` と `.codex/skills/` を更新する。`CLAUDE.md` はClaude Code側の引き継ぎ資料なので、**書き換えず残す**(内容が食い違ったらユーザーに確認)
- 記事や機能の作りかけ(draft: true等)を見つけても、勝手に完成・削除しない。ユーザーに確認する

## 作業の進め方

- ユーザーは非エンジニアの個人事業主。実装・起動・確認まで全面委任スタイルで自走する
- 開発サーバーを起動し、ブラウザで見える状態にしてから判断を仰ぐ
- 細かい確認で止まらない。重大な判断(公開・お金・会社関連の認証・データの場所)だけ質問する
- サイト改善は才流(sairu.co.jp)のメソッド準拠: ファーストビュー内CTA+実績明示、記事は前半テキストCTA+末尾リッチCTA、フォームは必須項目最小限

## 環境

- 作業場所は必ず `D:\dev\giken`。Google Drive(G:)上で npm を実行しない(npmが壊れる)
- Node.js はポータブル導入。シェルで見つからない場合: `$env:Path += ";$env:LOCALAPPDATA\Programs\nodejs"`
- 開発サーバー: `npm run dev`(ポート4321)。ビルド: `npm run build`(全ページ成功すること)
- GitHub は kuroroji/giken のみ。会社組織 cryptul には絶対に触れない
- **mainへのプッシュ=本番公開**(Cloudflare Pagesが自動デプロイ)。公開はユーザーの明示的な承認を得てから行う

## 文体・表記(全記事共通)

- 一人称は「僕」。です・ます調。実況者本人が話しているような、率直で構えない文体
- 見出しに「|」(パイプ)を使わない。「〜とは?〜!」のような口語的な区切りを使う
- タイトルの「!」「?」は半角(サイト全体で統一済み)
- ダッシュ「——」は使わない
- 商用・プロモーション文脈で「インディー」と書かない(無償依頼を呼び込むため)。「意欲作」「個人開発」などで言い換える
- AIが書いたような抽象的な文章、同じ主張の言い換え反復、SEO目的の水増しをしない
- 重要な箇所は太字にする。流し読みで太字だけ追えば要点が伝わる密度が目安

## 日本語Markdownの太字の罠(必読)

CommonMarkの仕様上、日本語文中の `**` は位置によって効かない。記事を編集したら必ずレンダリング確認する。

- NG: 閉じ `**` の直後に文字が続く(`**〜です。**ここから` は不可 → 改行か段落分けする)
- NG: 閉じ `**` の直前が文字で直後も文字(`**〜項目**です` は不可 → `**〜項目です**。` にする)
- NG: 開き `**` の直前が文字で直後が「 などの約物(`欄に**「ボタン」**` は不可 → 直前に読点を置く `欄に、**「ボタン」〜です。**`)
- 安全パターン: 太字を「読点・句点の直後から始め、句点の直前(または段落末)で閉じる」

## 表現の安全ルール(営業・案件系コンテンツ)

- 納期を保証しない。「内容により個別に案内。シンプルな内容なら数日〜1週間程度で対応できる場合もある」の形
- 成果を断定しない。「効果が見込める場合があります」「有効な選択肢になります」など条件付きで書く
- 報酬とゲーム評価が結びついているように見せない(「報酬でゲームへの評価や感想を変えるわけではない」を明確に)
- 実況者全体を代表する断定をしない。「仕事として案件を受けている実況者であれば〜と思います」のように限定する
- 案件系記事は「仕事として依頼を受けている実況者の一人である僕の場合の話」という前提を明示する
- 実績数値(ROAS等)は実測値のみ・注記付き。「絶対」「必ず成果が出る」は禁止

## 公開前の確認工程

1. `npm run build` が全ページ成功すること(frontmatterのZodスキーマ検証を兼ねる)
2. 開発サーバーで対象ページを開き、本文に生の `**` が残っていないこと・太字が効いていることを確認
3. 記事内リンクは実在するページのみ(`/promotion/`、`/profile/`、`/contact/` など)
4. コミットメッセージは日本語で変更内容を要約
5. ユーザーの承認を得てから `git push origin main`
6. プッシュ後、本番URL(https://kuroroji.site)に変更が反映されたことを `curl` 等で確認して報告する
